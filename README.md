# Dynamic Pricing and Sales Anomaly Detection

Прототип ML-системы для обнаружения аномалий в продажах, объяснения причин
изменения спроса и расчёта цен на следующий день. Решение подготовлено для
данных ритейлера: 200 товаров, ежедневная история за 2022–2023 годы и 146 000
наблюдений.

Основной аналитический сценарий находится в `Middle_MLE_HW.ipynb`, а
переиспользуемая логика разделена по небольшим предметным модулям пакета
`retail_pricing/`.

## Что реализовано

1. Аудит качества данных и проверка временной панели.
2. Past-only детектор технических всплесков и неожиданных нулевых продаж.
3. Оценка детектора через `Average Precision`, precision/recall и
   `precision@K`.
4. Глобальная XGBoost-модель спроса с причинными временными признаками.
5. Expanding-window валидация и сравнение с rolling baseline.
6. Root Cause Analysis на основе встроенного XGBoost TreeSHAP.
7. Оценка и регуляризация ценовой эластичности по товарам и сегментам.
8. Подбор цен для максимизации продаж, GMV и прибыли с бизнес-ограничениями.
9. Сохранение рекомендаций и диагностических таблиц в CSV.
10. Воспроизводимый запуск локально и в Docker.

## Данные и постановка

Панель содержит ежедневные наблюдения по 200 товарам с 1 января 2022 года по
31 декабря 2023 года. Целевая переменная — `sales`. Среди доступных факторов:
цена, себестоимость, промо, цена конкурента, рекламные расходы, остаток,
погодный индекс, категория и идентификатор товара.

Решаются две связанные задачи:

- найти подозрительные значения продаж и объяснить контекст события;
- спрогнозировать спрос на следующий день и выбрать цену под заданную
  бизнес-цель.

Целевые функции разделены явно:

```text
Sales(p)  = predicted demand
GMV(p)    = p * predicted demand
Profit(p) = (p - cost) * predicted demand
```

GMV и прибыль не считаются одним и тем же показателем: их оптимальные цены
могут различаться.

## 1. Детекция аномалий

Для каждого товара рассчитываются локальные статистики только по предыдущим
28 дням. Это не допускает использования будущих наблюдений при выставлении
алерта. Оценка всплеска нормируется на локальный разброс и уровень спроса, а
порог калибруется expanding-способом только по уже известной истории.

Детектор разделён на два уровня:

- `high_confidence` — технические всплески с высокой точностью;
- `review` — более широкий список, включающий неожиданные нулевые продажи.

Обычный рост продаж во время промо не считается технической ошибкой. Он
попадает в high-confidence слой только при экстремальном превышении порога.

### Качество детектора

| Метрика | Значение |
|---|---:|
| Average Precision | 0.8785 |
| ROC AUC | 0.9344 |
| High-confidence precision | 1.0000 |
| High-confidence recall | 0.8000 |
| Review precision | 0.3273 |
| Review recall | 0.9000 |
| precision@20 | 0.8500 |
| recall@20 | 0.8500 |
| precision@50 | 0.3600 |
| recall@50 | 0.9000 |

`Average Precision` и `precision@K` здесь информативнее accuracy, поскольку
аномалии редки. Например, `precision@20 = 0.85` означает, что среди 20 первых
событий рейтинга 17 соответствуют разметке аномалий.

### Наблюдения

- Главным источником ложных алертов простого spike-detector оказались
  промо-дни. Добавление promo-контекста подняло Average Precision примерно с
  0.80 до 0.88.
- High-confidence слой подходит для автоматического алерта: в offline-разметке
  его precision равен 1.0, но он сознательно пропускает часть событий.
- Review-слой находит 90% размеченных аномалий, однако требует ручной проверки.
- Нулевые продажи сложнее интерпретировать без журналов остатков, поставок и
  ETL-сбоев. Поэтому они не объявляются технической ошибкой автоматически.

## 2. Прогноз спроса

Используется одна глобальная XGBoost-модель для всех товаров. Такой подход
позволяет делиться статистической силой между рядами, сохраняя различия через
`item_id`, категорию и контекстные признаки.

Модель обучается непосредственно на `sales` с objective
`reg:absoluteerror`. Эта цель согласована с MAE и WAPE, а абсолютная ошибка
устойчивее квадратичной к редким экстремальным продажам. На одинаковых
временных фолдах она оказалась лучше предыдущего варианта с target
`log1p(sales)`.

Основные признаки:

- лаги 1, 2, 3, 7, 14, 21 и 28 дней;
- rolling mean/median/std и EWM;
- средний спрос по тому же дню недели;
- краткосрочный тренд и отношение недельных лагов;
- цена относительно себестоимости, конкурента и 90-дневной регулярной цены;
- история спроса в текущем promo-режиме;
- реклама, остаток, погода и календарь.

Аномальные строки исключаются из обучения, а при построении последующих лагов
подозрительное значение заменяется локальной медианой. Валидация состоит из
трёх expanding-window фолдов по 90 дней.

### Средние результаты временной валидации

| Модель | Scope | MAE | WAPE | SMAPE |
|---|---|---:|---:|---:|
| XGBoost | Все строки | 537.17 | 0.3477 | 0.4396 |
| XGBoost | Без review-кандидатов | 535.52 | 0.3470 | 0.4394 |
| Rolling mean 30 | Все строки | 558.87 | 0.3618 | 0.5235 |
| Rolling mean 30 | Без review-кандидатов | 557.23 | 0.3611 | 0.5233 |

XGBoost улучшает WAPE относительно baseline примерно на 3.9% и заметнее
улучшает SMAPE. Переход с `log1p(sales)` на прямую MAE-цель снизил WAPE с
0.3523 до 0.3477, а SMAPE с 0.4708 до 0.4396. При этом абсолютный WAPE около
35% показывает, что задача остаётся сложной: товары неоднородны, а продажи
имеют тяжёлый хвост.

## 3. Root Cause Analysis

Для объяснения прогноза используется встроенный TreeSHAP XGBoost через
`pred_contribs=True`. Отдельная библиотека `shap` не требуется, но сами
SHAP-вклады действительно рассчитываются и выражаются в единицах прогнозных
продаж.

Отчёт разделяет два вопроса:

1. Какой фактор сильнее всего сформировал ожидаемый спрос модели?
2. Насколько фактическое значение отклоняется от этого ожидания?

Если остаток на log-шкале слишком велик, событие получает причину
`unexplained_residual`. Это не позволяет ошибочно назвать крупный технический
выброс следствием фактора, который лишь немного изменил прогноз модели.

## 4. Эластичность

Индивидуальная ценовая эластичность оценивается Ridge-регрессией на log-шкале
с контролями промо, рекламы, конкурента, календаря и истории спроса. Затем
оценка регуляризуется к медиане категории и ограничивается диапазоном
от -5 до -0.05.

Получено:

- 107 товаров с индивидуально поддержанной оценкой `item_supported`;
- 93 товара с консервативной оценкой `category_prior_driven`.

### Наблюдения

- У части товаров сырая эластичность имеет положительный знак. Это не следует
  интерпретировать как причинный рост спроса от повышения цены: цена, промо и
  сезонный спрос меняются совместно.
- Для слабых индивидуальных оценок используется prior сегмента, а допустимое
  изменение цены сокращается до 5%.
- Даже высокий `R2` локальной регрессии не устраняет observational confounding.
  Надёжная причинная эластичность требует ценового эксперимента.

## 5. Оптимизация цены

Для следующего дня строится консервативный сценарий без промо. Будущие внешние
признаки заменяются последними устойчивыми значениями. Для каждой цены сетки
рассчитываются ожидаемые продажи, GMV и прибыль.

Точечная эластичность часто приводит к оптимуму на границе допустимого
диапазона. Поэтому profit-рекомендация строится robust-способом: для каждой
цены рассматриваются production-эластичность, индивидуальная sign-constrained
оценка и peer prior, после чего максимизируется худшая прибыль среди этих
сценариев.

Guardrails:

- цена не ниже `cost * 1.05`;
- изменение не более 15% для `item_supported`;
- изменение не более 5% для `category_prior_driven`;
- учёт 5–95% диапазона исторических регулярных цен, если ограничения имеют
  непустое пересечение;
- удержание цены при почти нулевом прогнозном спросе;
- сохранение текущей цены, если лучший worst-case вариант не даёт хотя бы 1%
  расчётного прироста прибыли;
- явная маркировка конфликтов ограничений.

### Итоговые рекомендации для 200 товаров

| Статус | Товаров | Интерпретация |
|---|---:|---|
| `robust_optimized` | 53 | Цена максимизирует worst-case прибыль и проходит uplift-порог |
| `insufficient_robust_uplift_hold` | 56 | Robust uplift ниже 1%, текущая цена сохранена |
| `low_demand_hold` | 61 | Цена удержана из-за низкого прогнозного спроса |
| `historical_range_reentry` | 10 | Цена возвращается в допустимый исторический диапазон |
| `historical_range_conflict_hold` | 12 | Текущая цена конфликтует с историческим диапазоном |
| `margin_floor_correction` | 8 | Цена поднята до минимально допустимой маржи |

Цена фактически меняется у 71 из 200 товаров. Среднее абсолютное изменение
profit-optimal цены составляет 3.03%. Для 53 строк `robust_optimized` медианный
worst-case uplift равен 12.66%, а минимальный — 1.40%.

Эти значения рассчитаны моделью и не являются доказанным финансовым uplift.
Порог 1% — настраиваемый бизнес-параметр, который в production должен учитывать
стоимость смены цены, требуемую статистическую мощность и риск ошибки
эластичности.

## Ограничения решения

- Offline-данные не доказывают причинный эффект цены.
- Метрики аномалий используют предоставленную скрытую разметку и не гарантируют
  такое же качество на новых периодах.
- Сценарий следующего дня предполагает отсутствие промо и использует
  приближения для неизвестных будущих факторов.
- `stock_level` используется как признак, но не как жёсткий cap: в исходных
  данных продажи иногда превышают указанный остаток.
- Оптимизация использует модель постоянной эластичности и не учитывает
  каннибализацию между товарами.
- Финансовый uplift нельзя оценить по той же истории, на которой построена
  политика.

## Как переносить в production

1. Запускать рекомендации в shadow mode без автоматической смены цен.
2. Контролировать качество входных данных и freshness признаков.
3. Мониторить WAPE, drift, долю алертов и срабатывания guardrails.
4. Провести ограниченный A/B-тест с рандомизацией по товарам или магазинам.
5. Измерять profit uplift, GMV, продажи, маржу и негативные побочные эффекты.
6. Расширять диапазон изменения цены только после подтверждения устойчивости.

## Итог

Решение закрывает полный offline-контур: от проверки данных и обнаружения
сбоев до прогноза спроса, объяснения событий и формирования безопасных ценовых
рекомендаций. Главный практический вывод состоит в том, что качество прогноза
само по себе недостаточно для автоматического ценообразования: необходимы
консервативные guardrails, причинный эксперимент и production-мониторинг.
